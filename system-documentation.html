How the two sides connect

Both sides share the same data layer — eBay API and TCG API. The deal finder reads from eBay to find opportunities. The listing agent writes to eBay to publish inventory. Same APIs, different directions.

Deal algorithm — cost model

Every listing is run through this formula. You configure the fee rates. The result is the true net profit if you bought and resold the card.

ComponentExample
eBay listing price$40.00
− eBay seller fees (~13.25%)−$5.30
− Shipping cost (est.)−$4.50
− Payment processing (~3%)−$1.20
− Your purchase price−$18.00
= Net profit estimate+$11.00 (61% margin)

Only cards above your configured minimum margin threshold (e.g. 25%) are flagged as deals. You tune this over time.

The listing skill — how it works

The skill is a prompt template your app fills in for every listing. Claude receives structured data and returns structured output. Here's what the prompt payload looks like:

// System prompt (your skill — stays constant)
You are an expert eBay trading card seller. Generate optimized listings
that rank well and convert. Follow eBay best practices for titles.
Use the sold comps as style reference — match their tone and structure.
Output JSON with: title, price, description, item_specifics.
// User message (filled dynamically per card)
card: 2003 Pokemon EX Dragon Flygon #94 Holo Rare
condition: Near Mint
tcg_market_price: $47.00
tcg_low_price: $34.00

// Sold comps injected here — these are the "training examples"
sold_comps: [
{ title: "2003 Pokemon EX Dragon Flygon 94/97 Holo PSA 9",
sold_price: "$89", description: "..." },
{ title: "Pokemon Flygon Holo Rare EX Dragon NM 94/97 2003",
sold_price: "$52", description: "..." }
]
// Claude returns structured JSON your app parses
→ title: "2003 Pokemon EX Dragon Flygon #94 Holo Rare NM"
→ price: 49.99
→ description: "Up for sale is a Near Mint..."
→ item_specifics: { set: "EX Dragon", year: "2003", ... }

Over time, you curate your best-performing listings and add them to the skill as permanent examples. The skill gets better as your example library grows — no retraining, no fine-tuning needed.

Claude API — model selection guide

Different card types warrant different models. Haiku is fast and cheap for routine listings. Opus is worth the cost for high-value cards where a richer description meaningfully improves conversion.

Use caseModelCost/listingWhen to use
Bulk commons & uncommons Haiku 4.5 ~$0.002 Cards under $5. Volume listings, simple templates fine too.
Standard singles & holos Sonnet 4.6 ~$0.01 $5–$100 cards. Most of your inventory. Best quality/cost ratio.
Graded slabs & vintage Opus 4.7 ~$0.05 $100+ cards. PSA/BGS slabs, 1st editions. Description quality matters.
Architecture evolution

Now vs. Phase 3 — what MCP changes

The v1 app orchestrates all data fetching and passes results to Claude. In Phase 3, an MCP server gives Claude the ability to call APIs itself, shifting it from a passive text generator to an active agent.

Phase 1 & 2 — App orchestrates
Your app calls eBay API, fetches listings
Your app calls TCG API, gets pricing
Your app bundles everything, calls Claude
Claude generates listing text only
Your app submits the listing to eBay
You review at each step manually
Phase 3 — Claude orchestrates via MCP
You trigger Claude with a card name
Claude calls eBay API tool itself
Claude calls TCG API tool itself
Claude generates and prepares the listing
Claude calls eBay Sell API tool to publish
You only review the final draft before it goes live

MCP is the right endgame but adds real infrastructure complexity. Build and validate the manual version first — you'll understand the edge cases far better before automating them away.
